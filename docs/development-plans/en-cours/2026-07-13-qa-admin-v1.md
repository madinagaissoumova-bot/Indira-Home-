# Plan - QA Admin V1

## Objectif

Verifier et corriger le parcours admin V1 afin que la vendeuse puisse se connecter, gerer les produits, categories, stocks et commandes sans exposer de donnees sensibles ni casser les regles metier.

## Validation Du Plan

- [x] Le plan a ete propose apres le merge du Plan 2.
- [x] Le perimetre a ete valide par l'utilisatrice.
- [x] Une branche dediee a ete creee depuis `main`.
- [x] Ce fichier a ete enregistre dans `docs/development-plans/en-cours/`.

## Perimetre

- Tester la protection des routes admin et la connexion/deconnexion.
- Tester le dashboard admin et ses liens principaux.
- Tester la gestion produits : brouillon, publication incomplete refusee, publication complete, masquage, suppression autorisee ou bloquee.
- Tester la gestion categories et sous-categories : creation, masquage, affichage, suppression autorisee ou bloquee.
- Tester la gestion stock : ajout, retrait, correction, refus des valeurs invalides et effet cote client.
- Tester les commandes : liste, detail, note interne, changement de statut, annulation et remise en stock une seule fois.
- Verifier que les donnees personnelles des commandes restent uniquement dans l'admin protege.
- Corriger les bugs ou incoherences trouves dans ce perimetre.

## Hors Perimetre

- Ne pas modifier le design public sauf si une regression admin le rend necessaire.
- Ne pas ajouter de fonctionnalite hors V1.
- Ne pas changer les secrets ou valeurs reelles de production.
- Ne pas lancer de migration destructive.
- Ne pas merger sans validation explicite de l'utilisatrice.

## Verification

- `npm run lint`
- `npm run check:docs`
- `npm test`
- `npm run build`
- Verification HTTP des routes admin protegees.
- Verification Playwright desktop/tablette des pages admin critiques.
- Verification que les actions admin sans session sont refusees.

## Risques Et Contraintes

- Les tests admin peuvent ecrire en base locale ou Supabase de developpement ; eviter de polluer les donnees si ce n'est pas necessaire.
- Les commandes contiennent des donnees personnelles ; ne pas les exposer dans des logs publics ou dans des pages clientes.
- Les specs actuelles indiquent que l'annulation admin remet le stock en ligne une seule fois ; cette regle prime si un ancien document ou skill dit autre chose.
