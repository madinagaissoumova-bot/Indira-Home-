# Plan - QA finale V1

## Objectif

Verifier que la V1 Indira Home est coherente de bout en bout avant mise en ligne, sans ajouter de fonctionnalites hors perimetre.

## Validation Du Plan

- [x] Le plan a ete propose apres le merge du plan admin QA.
- [x] Le perimetre a ete valide par l'utilisatrice.
- [x] Une branche dediee a ete creee depuis `main`.
- [x] Ce fichier a ete enregistre dans `docs/development-plans/en-cours/`.

## Perimetre

- Verifier les routes publiques principales : accueil, categories, recherche, favoris, panier, checkout, confirmation, a propos et confidentialite.
- Verifier les routes admin protegees : login, dashboard, produits, categories, stock et commandes.
- Verifier les criteres V1 : textes clientes en russe, pas de paiement en ligne, pas de comptes clientes, livraison limitee a la Republique tchetchene.
- Verifier que les donnees personnelles de commande ne sont pas exposees cote public.
- Verifier les variables d'environnement attendues et l'absence de secrets commites.
- Verifier mobile, tablette et desktop sur les pages critiques.
- Corriger uniquement les petits bugs bloquants ou incoherences trouves dans ce perimetre.

## Hors Perimetre

- Ne pas ajouter de nouvelle fonctionnalite.
- Ne pas faire de gros redesign.
- Ne pas lancer de migration destructive.
- Ne pas modifier les secrets reels.
- Ne pas merger sans validation explicite de l'utilisatrice.

## Etapes

- [x] 1. Verifier `main`, les plans actifs et les specs.
- [x] 2. Auditer configuration, secrets, routes et surface publique.
- [x] 3. Tester les parcours publics critiques.
- [x] 4. Tester les protections admin critiques.
- [x] 5. Verifier responsive mobile, tablette et desktop.
- [x] 6. Corriger les bugs ou incoherences bloquants trouves.
- [x] 7. Lancer les checks adaptes.
- [x] 8. Presenter le resultat avant commit, Pull Request et merge.

## Verification

- `npm run prisma:generate`
- `npm run lint`
- `npm run check:docs`
- `npm test`
- `npm run build`
- Verifications HTTP locales des routes publiques et admin.
- Verifications Playwright des pages critiques sur mobile, tablette et desktop.

## Risques Et Contraintes

- Les tests d'integration base ne doivent pas polluer des donnees reelles sans accord explicite.
- Les secrets et donnees personnelles ne doivent pas etre affiches dans les logs ou les reponses.
- Les points go-live lies a la vraie production peuvent rester comme checklist si les valeurs production ne sont pas disponibles localement.
